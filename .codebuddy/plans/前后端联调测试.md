## 产品概述

前后端功能联调测试，确保所有页面功能正常运行。从数据库初始化开始，清空 SQLite 测试数据库，运行后端自动化测试用例（含 initdb/seedrbac/createsuperuser 流程验证），启动前后端服务，通过浏览器逐页面操作测试所有功能点，发现并修复所有问题，最后验证前端生产打包流程，确保项目可正常发布到生产环境。

## 核心功能

- **阶段一：数据库初始化验证**
  - 清空 SQLite 测试数据库（删除 `service/sql/test.db`）
  - 执行 `python -m scripts.cli initdb` 验证建表流程
  - 执行 `python -m scripts.cli seedrbac` 验证 RBAC 种子数据初始化
  - 执行 `python -m scripts.cli seeddata` 验证测试数据初始化
  - 执行 `python -m scripts.cli createsuperuser` 验证超级管理员创建
  - 或一键执行 `python -m scripts.cli initall` 验证完整初始化链路
  - 确认数据库表结构和约束正确（外键、唯一索引等）
  - 确认种子数据完整性（菜单树、角色-菜单关联、用户-角色关联）

- **阶段二：后端自动化测试**
  - 运行 `pytest tests/` 全部测试，确保所有单元测试和集成测试通过
  - 单元测试使用内存 SQLite（conftest.py 自动创建/销毁表）
  - 集成测试使用共享内存 SQLite + db_seed 清空再种子数据
  - 重点关注集成测试中覆盖的 API 端点：认证、用户CRUD、角色菜单部门、日志
  - 检查 git 未提交的 IP 规则相关改动是否影响现有功能

- **阶段三：前端 UI 测试（使用 Playwright MCP）**
  - 使用 `[mcp:Playwright MCP Server]` 进行浏览器自动化测试
  - 逐页面访问，验证每个功能按钮和表单操作
  - 按优先级测试：登录 → 用户管理 → 角色管理 → 部门管理 → 菜单管理 → 字典管理 → IP规则 → 系统配置 → 日志 → 在线用户

- **阶段四：问题修复**
  - 根据测试结果分类：前端bug / 后端bug / 前后端对接bug
  - 优先修复阻断性问题（页面无法加载、API 500 错误）
  - 然后修复功能性问题（数据不显示、操作不生效）
  - 最后处理体验性问题（提示不友好、样式异常）

- **阶段五：前端生产打包验证**
  - 执行 `pnpm build` 验证生产环境打包是否成功
  - 检查打包产物完整性（dist 目录结构、静态资源、入口文件）
  - 执行 `pnpm typecheck` 验证 TypeScript 类型检查通过
  - 执行 `pnpm lint` 验证代码规范检查通过
  - 使用 `pnpm preview` 启动生产预览服务，验证打包产物可正常运行
  - 验证生产环境下 API 代理配置正确（Vite proxy 仅开发环境生效，生产需 nginx/网关代理）
  - 验证环境变量和平台配置文件（`public/platform-config.json`）在生产包中正确
  - 验证路由模式（history/hash）在生产环境下正常工作
  - 检查打包体积是否合理，无异常大的 chunk
  - 确认 WASM/Worker 等特殊资源（`public/wasm/`）正确复制到 dist
  - 确认字体、图标等静态资源正确打包（无 404 或加载失败）
  - 确认所有动态路由和权限控制在生产包中正常工作

- **逐页面测试清单**：
  - 用户管理：列表/新增/编辑/删除/批量删除/重置密码/状态修改/分配角色
  - 角色管理：列表/新增/编辑/删除/状态修改/菜单权限分配
  - 部门管理：列表/新增/编辑/删除
  - 菜单管理：列表/新增/编辑/删除/菜单树
  - 字典管理：列表/按名称查询/新增/编辑/删除
  - IP规则管理：列表/新增/编辑/删除/批量删除/清空
  - 系统配置：列表/新增/编辑/删除
  - 登录日志：列表/批量删除/清空
  - 操作日志：列表/批量删除/清空
  - 系统日志：列表/查看详情
  - 在线用户：列表/强制下线

## 技术栈

- 后端: FastAPI + SQLModel + SQLAlchemy 2.0 (AsyncIO) + pytest
- 前端: Vue 3 + TypeScript + Element Plus + Pure Admin
- 测试工具: pytest (后端), Playwright MCP (前端UI测试)
- HTTP客户端: Axios
- 数据库: SQLite (测试环境) / MySQL (开发环境)

## 实施方案

### 方案概述

分五个阶段推进：1) 数据库初始化验证 → 确保建表和种子数据正确；2) 后端自动化测试 → 修复后端问题；3) 前端服务启动+逐页面UI测试 → 记录问题；4) 集中修复所有前后端联调问题；5) 前端生产打包验证 → 确保项目可正常发布到生产环境。

### 测试策略

**阶段一：数据库初始化验证（从零开始）**

关键步骤：
1. 删除旧测试数据库 `service/sql/test.db`（如果存在）
2. 设置 `APP_ENV=testing` 使用 SQLite
3. 执行 `python -m scripts.cli initall` 一键初始化，验证：
   - 建表成功（检查 SQLModel.metadata.create_all 无报错）
   - RBAC 种子数据完整（DEFAULT_MENUS/DEFAULT_ROLES 正确写入）
   - admin 角色分配了所有菜单权限
   - user 角色分配了只读菜单权限
   - 超级管理员创建成功（username=admin, 自动分配 admin 角色）
4. 验证数据库表结构和约束：
   - `sys_users` 表存在，admin 用户 is_superuser=1
   - `sys_roles` 表存在，admin 和 user 角色已创建
   - `sys_menus` + `sys_menumeta` 表存在，菜单树完整
   - `sys_userinfo_roles` 关联正确
   - `sys_userrole_menu` 关联正确（admin 角色有所有菜单）
   - `sys_departments` 表存在
   - `sys_ip_rules` 表存在
   - `sys_systemconfig` 表存在
   - `sys_loginlog` 表存在（种子日志数据）
   - `sys_logs` 表存在（种子系统日志数据）
5. 如初始化过程有报错，定位并修复后再重试

**阶段二：后端测试**

- 运行 `pytest tests/ -v` 全部测试
- 单元测试（7个文件）：test_auth, test_core, test_dto_validators, test_menu_service, test_role_service, test_system_config_service, test_user_service, test_validators
- 集成测试（2个文件）：
  - test_api.py — 基础健康检查/认证/用户端点
  - test_api_real_flow.py — 全链路真实流程（认证/RBAC/用户CRUD/角色菜单部门/日志/桩路由）
- 集成测试每个用例前自动清空表 + 插入种子数据（db_seed.py）
- 检查 IP 规则相关改动（新增 get_effective_ip_rules、缓存方法）是否破坏现有功能

**阶段三：前端 UI 测试（使用 Playwright MCP）**

- 使用 `[mcp:Playwright MCP Server]` 进行浏览器自动化测试
- 逐页面访问，验证每个功能按钮和表单操作
- 按优先级测试：登录 → 用户管理 → 角色管理 → 部门管理 → 菜单管理 → 字典管理 → IP规则 → 系统配置 → 日志 → 在线用户

**阶段四：问题修复**

- 根据测试结果分类：前端bug / 后端bug / 前后端对接bug
- 优先修复阻断性问题（页面无法加载、API 500 错误）
- 然后修复功能性问题（数据不显示、操作不生效）
- 最后处理体验性问题（提示不友好、样式异常）

**阶段五：前端生产打包验证**

- 执行 `pnpm build` 验证生产打包成功，无编译错误和警告
- 执行 `pnpm typecheck` 确保 TypeScript 类型检查通过（零错误）
- 执行 `pnpm lint` 确保代码规范检查通过（零错误）
- 检查 `dist/` 目录结构完整性：
  - `index.html` 入口文件存在且引用正确
  - `assets/` 目录包含 JS/CSS/图片等静态资源
  - `platform-config.json` 配置文件已复制
  - `wasm/` 目录（capture.worker.js、index.js）已复制
  - 字体和图标资源已正确打包
- 使用 `pnpm preview` 启动生产预览，验证：
  - 首页可正常加载，无白屏
  - 登录流程正常
  - 动态路由和菜单正常渲染
  - API 请求正常（需后端服务运行，或配置 nginx 代理）
  - 静态资源无 404 错误
- 验证生产环境配置：
  - Vite proxy 仅开发环境生效，生产环境需通过 nginx/网关转发 `/api/*`
  - 环境变量（VITE_*）在生产包中正确注入
  - 路由模式（history/hash）在生产环境正常
  - 打包体积合理，无异常大的 chunk（关注单个 chunk > 500KB 的情况）
- 如打包失败或运行异常，定位并修复后重新验证

### 关键技术决策

- 从数据库初始化开始验证，确保不遗漏任何建表/种子数据场景
- 后端测试优先于前端测试，确保 API 正确后再进行 UI 测试
- 使用 Playwright MCP 进行 UI 自动化测试，而非纯手动操作，确保测试可复现
- 问题修复采用"发现一批、修复一批、验证一批"的迭代方式
- SQLite 测试数据库在初始化验证后需保留，供前端联调时后端使用
- 联调验证全部通过后，进行前端生产打包验证，确保整个开发到发布流程闭环

## 实施要点

- 后端测试使用 SQLite 内存数据库（conftest.py 自动管理），不依赖外部服务
- 前端联调需要后端服务运行中，Vite 代理 /api/* 到 localhost:8000
- 前端联调后端使用 MySQL（.env.development），需确保 MySQL 服务可用
- 数据库初始化验证使用 SQLite（.env.testing），与后端测试保持一致
- 前端 HTTP 拦截器已实现 401 处理（`isLoggingOut` 标志位），需验证 token 过期场景
- IP 规则 API 前端仍为内联方式调用，需确认功能正常
- 部分监控页面（在线用户、地图、卡片列表）为 stub 数据，功能有限
- `python -m scripts.cli initall` = initdb + seedrbac + seeddata + createsuperuser 四步合一
- 清空测试数据库时直接删除 test.db 文件即可，initdb 会重新创建
- 前端生产打包验证需在所有联调问题修复后进行，确保打包基于无问题代码
- 生产环境 API 代理需通过 nginx 或 API 网关实现，Vite proxy 仅开发环境生效
- 打包验证重点关注：类型检查通过、lint 通过、构建无错误、预览可正常运行

## MCP

- **Playwright MCP Server**
- Purpose: 前端 UI 自动化测试，逐页面操作验证所有功能点
- Expected outcome: 覆盖所有页面的增删改查操作，截图记录问题，验证修复结果

## SubAgent

- **code-explorer**
- Purpose: 在测试过程中定位问题代码，搜索相关文件和依赖关系
- Expected outcome: 快速定位 bug 根因，辅助修复决策
