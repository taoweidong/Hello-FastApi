# Commit Changelog - 提交前自动记录修改

## 概述

此技能在每次 Git 提交前自动执行，总结本次代码变更内容并在 `README.md` 的「修改记录」章节中追加一条结构化的修改记录。记录按时间倒序排列，逐步累积，最终随代码一起提交入库。

## 触发条件

当用户表达以下意图时触发此技能：

- 准备提交代码（如"提交"、"commit"、"上库"）
- 要求记录本次修改（如"记录修改"、"更新修改记录"）
- 在 git commit 之前需要总结变更

## 工作流程

### 第一步：收集变更信息

执行以下命令获取本次变更的完整信息：

```bash
# 查看当前工作区和暂存区的变更概览
git status

# 查看已暂存变更的详细差异
git diff --cached --stat

# 查看未暂存变更的详细差异
git diff --stat

# 查看完整差异内容（用于理解具体修改）
git diff
git diff --cached
```

如果工作区是干净的（无变更），提示用户当前没有需要记录的修改。

### 第二步：分析并总结变更

根据 git diff 的结果，按以下维度总结修改内容：

1. **影响范围**：涉及哪些模块/目录（如后端 service/、前端 web/、配置文件等）
2. **修改类型**：新增功能(feat)、修复问题(fix)、重构(refactor)、文档(docs)、样式(style)、测试(test)、构建(chore)
3. **具体变更**：用简洁的中文描述每个重要变更点，避免过于技术化的细节

总结格式示例：
- 新增用户管理模块的批量删除功能
- 修复登录页面在移动端的样式问题
- 重构权限校验中间件，提取公共逻辑

### 第三步：获取修改人信息

```bash
# 获取当前 git 用户名
git config user.name
```

### 第四步：更新 README.md

在 `README.md` 文件中找到「修改记录」章节（`## 修改记录`），在其下的表格中追加一行新记录。

**记录格式**（Markdown 表格，按时间倒序排列，最新记录在最前面）：

```markdown
## 修改记录

| 时间 | 修改人 | 主要修改内容 |
|------|--------|-------------|
| 2026-04-26 09:20 | admin | 新增用户批量删除功能；修复移动端登录样式问题 |
| 2026-04-25 14:30 | dev | 重构权限校验中间件；添加单元测试 |
```

**规则**：

- 时间格式：`YYYY-MM-DD HH:mm`（使用当前系统时间）
- 修改人：使用 git config user.name 的值
- 主要修改内容：用分号分隔多个变更点，每条控制在 50 字以内
- 新记录始终插入表格的第一行（紧跟表头之后）
- 如果 `README.md` 中不存在「修改记录」章节，在「开发规范」章节之前自动创建

### 第五步：确认并提交

完成 README.md 更新后：

1. 展示即将写入的修改记录内容，让用户确认
2. 用户确认后，将 README.md 的更新加入暂存区：`git add README.md`
3. 提示用户可以继续执行 git commit

## 注意事项

- 只记录有意义的变更，忽略纯格式化、空行调整等无实质影响的修改
- 如果本次变更只涉及 README.md 自身的修改记录更新，则无需再次追加记录
- 修改内容的描述应面向团队成员，使用业务语言而非技术实现细节
- 如果暂存区和工作区都有变更，需要合并分析，以暂存区为准（因为 commit 只提交暂存区内容）
