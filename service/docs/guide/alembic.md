# Alembic 数据库迁移指南

本文档介绍如何在 Hello-FastApi 项目中使用 Alembic 进行数据库迁移管理，包括初始化、创建迁移、应用迁移和回退操作。

[← 返回首页](../../README.md)

---

## 概述

项目使用 [Alembic](https://alembic.sqlalchemy.org/) 作为数据库迁移工具，配合 SQLModel（SQLAlchemy 的封装）实现版本化的数据库 Schema 管理。

### 当前配置

| 配置项 | 值 |
|--------|-----|
| 迁移目录 | `service/alembic/` |
| 版本文件目录 | `service/alembic/versions/` |
| 配置文件 | `service/alembic.ini` |
| 环境脚本 | `service/alembic/env.py` |
| 元数据来源 | `SQLModel.metadata` |
| 类型变更检测 | `compare_type=True`（已启用） |

### 数据库支持

项目通过 `DATABASE_URL` 环境变量支持多种数据库，Alembic 已针对不同数据库做了适配：

| 数据库 | 连接字符串格式 | 驱动 |
|--------|---------------|------|
| SQLite（默认） | `sqlite+aiosqlite:///./sql/dev.db` | aiosqlite |
| PostgreSQL | `postgresql+asyncpg://user:pass@localhost/dbname` | asyncpg |
| MySQL | `mysql+aiomysql://user:pass@localhost/dbname` | aiomysql |

> **SQLite 特殊处理**：`env.py` 会自动将 `sqlite+aiosqlite://` 替换为 `sqlite://` 使用同步模式执行迁移，因为 Alembic 的某些操作（如 batch 模式）在异步 SQLite 下不完全兼容。

## 前置条件

确保已安装项目依赖（`alembic >= 1.13.0` 已包含在依赖中）：

```bash
cd service
uv pip install -e ".[dev]"
```

确保 `service/` 目录下存在对应环境的 `.env` 文件，其中 `DATABASE_URL` 配置正确：

```bash
# .env.development（默认）
DATABASE_URL=sqlite+aiosqlite:///./sql/dev.db
```

## 常用命令

所有命令均需在 `service/` 目录下执行。

### 查看当前状态

```bash
# 查看当前迁移版本
alembic current

# 查看迁移历史
alembic history

# 以详细模式查看历史（含描述）
alembic history --verbose
```

### 创建迁移

```bash
# 自动检测模型变更，生成迁移脚本
alembic revision --autogenerate -m "描述信息"

# 示例：添加新表
alembic revision --autogenerate -m "add_notification_table"

# 示例：修改字段
alembic revision --autogenerate -m "add_email_to_users"

# 手动创建空迁移脚本（不自动检测变更）
alembic revision -m "custom_migration"
```

### 应用迁移

```bash
# 升级到最新版本
alembic upgrade head

# 升级到指定版本
alembic upgrade 001

# 升级一个版本（逐步升级）
alembic upgrade +1
```

### 回退迁移

```bash
# 回退到上一个版本
alembic downgrade -1

# 回退到指定版本
alembic downgrade 001

# 回退所有迁移（回到空数据库状态）
alembic downgrade base

# 回退两个版本
alembic downgrade -2
```

### 生成 SQL 脚本（不执行）

```bash
# 生成升级到最新版本的 SQL（不连接数据库）
alembic upgrade head --sql

# 生成从当前版本到指定版本的 SQL
alembic upgrade 002 --sql

# 生成回退一个版本的 SQL
alembic downgrade -1 --sql
```

### 其他

```bash
# 查看迁移脚本的详细内容（不执行）
alembic show 001

# 标记当前数据库为某个版本（不执行迁移，仅更新版本号）
alembic stamp head

# 清除所有版本记录（危险操作，不删除表数据）
alembic stamp base
```

## 完整工作流程

### 场景一：新环境初始化

在新环境或全新数据库上，应用所有迁移以创建完整 Schema：

```bash
cd service

# 1. 确认数据库配置
cat .env.development
# DATABASE_URL=sqlite+aiosqlite:///./sql/dev.db

# 2. 应用所有迁移
alembic upgrade head

# 3. 确认迁移状态
alembic current
# 应显示当前最新版本号

# 4. 种子数据（如需要）
python -m scripts.cli seedrbac
python -m scripts.cli createsuperuser -u admin -e admin@example.com -p admin123
```

### 场景二：修改模型并生成迁移

当修改了 `src/infrastructure/database/models/` 中的 SQLModel 模型后：

```bash
cd service

# 1. 修改模型代码（例如给 User 添加 email 字段）
#    编辑 src/infrastructure/database/models/user.py

# 2. 生成迁移脚本
alembic revision --autogenerate -m "add_email_to_users"

# 3. 检查生成的迁移脚本（重要！）
#    文件位于 alembic/versions/ 目录下
#    确认 upgrade() 和 downgrade() 内容正确

# 4. 应用迁移
alembic upgrade head

# 5. 验证
alembic current
```

### 场景三：回退错误的迁移

发现最近一次迁移有问题，需要回退：

```bash
cd service

# 1. 查看当前版本
alembic current
# 假设当前为 003，需要回退到 002

# 2. 回退一个版本
alembic downgrade -1

# 3. 确认版本
alembic current
# 应显示 002

# 4. 删除有问题的迁移文件
#    删除 alembic/versions/ 中对应的 .py 文件

# 5. 重新生成正确的迁移
alembic revision --autogenerate -m "fixed_email_field"

# 6. 应用新迁移
alembic upgrade head
```

### 场景四：回退到指定版本

```bash
# 回退到版本 001
alembic downgrade 001

# 从版本 001 直接升级到最新
alembic upgrade head

# 回退所有迁移（清空所有表）
alembic downgrade base
```

## 迁移脚本编写规范

### 自动生成的迁移脚本

使用 `--autogenerate` 时，Alembic 会对比 `SQLModel.metadata` 与实际数据库的差异，自动生成 `upgrade()` 和 `downgrade()` 函数。

生成的文件示例：

```python
"""add email to users

Revision ID: 002
Revises: 001
Create Date: 2026-05-05 21:00:00.000000
"""

from alembic import op
import sqlalchemy as sa
import sqlmodel

# revision identifiers, used by Alembic.
revision = "002"
down_revision = "001"
branch_labels = None
depends_on = None


def upgrade() -> None:
    # ### commands auto generated by Alembic - please review! ###
    op.add_column("sys_users", sa.Column("email", sqlmodel.AutoString(length=255), nullable=True))
    # ### end Alembic commands ###


def downgrade() -> None:
    # ### commands auto generated by Alembic - please review! ###
    op.drop_column("sys_users", "email")
    # ### end Alembic commands ###
```

### 手动编写注意事项

1. **务必实现 `downgrade()`**：每个迁移必须能回退，确保 `downgrade()` 正确还原 `upgrade()` 的变更
2. **注意外键依赖顺序**：删除表时先删依赖表，创建表时先建被依赖表
3. **SQLite 限制**：SQLite 不支持 `ALTER TABLE DROP COLUMN`（Alembic 的 batch 模式可绕过此限制，但需显式启用）
4. **使用 batch 模式（SQLite）**：如果项目使用 SQLite，对表的修改操作应使用 batch 模式：

```python
def upgrade() -> None:
    # SQLite 安全的表修改方式
    with op.batch_alter_table("sys_users") as batch_op:
        batch_op.add_column(sa.Column("email", sa.String(255), nullable=True))


def downgrade() -> None:
    with op.batch_alter_table("sys_users") as batch_op:
        batch_op.drop_column("email")
```

5. **数据迁移**：如果需要在迁移中处理数据，使用 `op.get_bind()` 获取连接：

```python
def upgrade() -> None:
    # 先改结构
    op.add_column("sys_users", sa.Column("status", sa.String(20), nullable=True))

    # 再迁移数据
    conn = op.get_bind()
    conn.execute(
        sa.text("UPDATE sys_users SET status = 'active' WHERE status IS NULL")
    )
```

6. **命名约束**：项目表名使用 `sys_` 前缀，命名约束时保持一致：

```python
# 外键命名约定：fk_{源表}_{目标表}
op.create_foreign_key("fk_sys_users_sys_departments", "sys_users", "sys_departments", ["dept_id"], ["id"])

# 索引命名约定：ix_{表名}_{列名}
op.create_index("ix_sys_users_email", "sys_users", ["email"], unique=True)
```

## 初始迁移说明

项目当前包含一个初始迁移 `001_initial_schema`，它使用 `SQLModel.metadata.create_all()` 创建了所有基础表：

- `sys_users` — 用户表
- `sys_roles` — 角色表
- `sys_menus` — 菜单表
- `sys_menumeta` — 菜单元信息表
- `sys_departments` — 部门表
- `sys_dictionary` — 字典表
- `sys_systemconfig` — 系统配置表
- `sys_userloginlog` — 登录日志表
- `sys_logs` — 操作日志表
- `sys_ip_rules` — IP 规则表
- `sys_userinfo_roles` — 用户-角色关联表
- `sys_userrole_menu` — 角色-菜单关联表

> **注意**：初始迁移使用 `create_all()` 而非 autogenerate，这是首次搭建时的常见做法。后续所有新迁移应使用 `alembic revision --autogenerate` 生成。

## 环境变量与多环境迁移

项目支持三种环境，通过 `APP_ENV` 切换，Alembic 通过 `env.py` 读取 `Settings()` 获取 `DATABASE_URL`：

| 环境 | `APP_ENV` 值 | 配置文件 | 数据库 |
|------|--------------|---------|--------|
| 开发 | `development` | `.env.development` | `sql/dev.db` |
| 测试 | `testing` | `.env.testing` | `sql/test.db` |
| 生产 | `production` | `.env.production` | 由配置决定 |

对生产环境执行迁移时，务必先在测试环境验证：

```bash
# 1. 在测试环境验证
APP_ENV=testing alembic upgrade head

# 2. 生成 SQL 预览
APP_ENV=production alembic upgrade head --sql > migration.sql

# 3. 确认无误后执行
APP_ENV=production alembic upgrade head
```

## 常见问题

### Q1: `alembic current` 报错找不到版本表

数据库尚未执行过迁移，运行 `alembic upgrade head` 即可。

### Q2: 已有数据库但无版本记录

如果数据库已通过 `create_all()` 或 `initdb` 创建了表，但 Alembic 没有版本记录：

```bash
# 标记当前数据库为最新版本（不执行任何操作，仅写入版本号）
alembic stamp head
```

### Q3: autogenerate 未检测到变更

可能的原因：
- 模型未被 `src/infrastructure/database/models/__init__.py` 导入
- `env.py` 中的 `import src.infrastructure.database.models` 缺失
- 变更已在数据库中存在

### Q4: SQLite 下 ALTER TABLE 报错

SQLite 对 ALTER TABLE 支持有限。使用 batch 模式：

```python
with op.batch_alter_table("sys_users") as batch_op:
    batch_op.alter_column("name", new_column_name="full_name")
```

### Q5: 迁移产生分歧（multiple heads）

```bash
# 查看所有头版本
alembic heads

# 合并分歧
alembic merge -m "merge_heads" <revision1> <revision2>
```

### Q6: 想要完全重置迁移

```bash
# 1. 回退到 base（删除所有表）
alembic downgrade base

# 2. 删除不需要的版本文件
rm alembic/versions/xxx_unwanted.py

# 3. 重新生成
alembic revision --autogenerate -m "fresh_start"

# 4. 应用
alembic upgrade head
```

## 最佳实践

1. **每次模型变更后立即生成迁移**，不要积累多个变更一次性生成
2. **始终检查自动生成的迁移脚本**，autogenerate 可能遗漏或误判
3. **确保 `downgrade()` 正确**，在开发环境中测试回退
4. **迁移脚本纳入版本控制**，与其他代码一起提交
5. **生产环境迁移前先备份**，并使用 `--sql` 预览
6. **避免修改已提交的迁移文件**，如需修改应新建迁移
7. **在 CI/CD 中验证迁移**，确保 `alembic upgrade head` 和 `alembic downgrade base` 均可正常执行
8. **新增模型务必在 `__init__.py` 中导入**，否则 Alembic 无法检测到新表
