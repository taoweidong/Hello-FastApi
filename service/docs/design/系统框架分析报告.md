# Hello-FastApi 后端系统框架分析报告

> 评估日期：2026-04-18
> 评估范围：`service/` 后端服务全栈架构、技术选型、DDD 实现、大型项目演进能力

---

## 一、项目概览

Hello-FastApi 是一个基于 FastAPI 的全栈管理系统后端，采用 DDD（领域驱动设计）四层架构，实现 JWT 双令牌认证和 RBAC 权限控制。项目当前覆盖用户管理、角色管理、菜单管理、部门管理、字典管理、日志审计、IP 规则、系统配置等 8 个核心业务模块。

### 技术栈总览

| 层次 | 技术选型 | 版本要求 |
|------|---------|---------|
| Web 框架 | FastAPI + classy-fastapi | >=0.115.0 |
| ASGI 服务器 | Uvicorn | >=0.30.0 |
| ORM | SQLModel (SQLAlchemy + Pydantic) | >=0.0.22 |
| CRUD 助手 | FastCRUD | >=0.15.0 |
| 数据库 | SQLite / MySQL / PostgreSQL | 多数据库支持 |
| 缓存 | Redis (async) | >=5.0.0 |
| 认证 | JWT (python-jose + bcrypt) | >=3.3.0 / >=4.0.0 |
| 配置管理 | pydantic-settings | >=2.0.0 |
| 日志 | Loguru | >=0.7.0 |
| HTTP 客户端 | httpx | >=0.27.0 |
| 包管理 | UV + hatchling | - |
| 测试 | pytest + pytest-asyncio + httpx | >=8.0.0 |
| 代码质量 | Ruff + MyPy | >=0.6 / >=1.11 |

## 二、架构分层分析

### 2.1 分层结构

```
┌─────────────────────────────────────────────────────────┐
│                    API 层 (src/api/)                     │
│  路由定义 │ 依赖注入 │ 统一响应 │ 认证授权              │
├─────────────────────────────────────────────────────────┤
│                 应用层 (src/application/)                │
│  应用服务 │ DTO │ 用例编排 │ 事务控制                    │
├─────────────────────────────────────────────────────────┤
│                  领域层 (src/domain/)                    │
│  实体 │ 仓储接口(ABC) │ 领域服务 │ 异常定义             │
├─────────────────────────────────────────────────────────┤
│              基础设施层 (src/infrastructure/)            │
│  ORM模型 │ 仓储实现 │ 数据库管理 │ 缓存 │ 中间件        │
└─────────────────────────────────────────────────────────┘
```

### 2.2 各层实现质量评估

| 层次 | 完成度 | 评价 |
|------|--------|------|
| API 层 | ★★★★☆ | 路由结构清晰，依赖注入完整，统一响应规范；但类路由模式与标准 FastAPI 生态兼容性一般 |
| 应用层 | ★★★☆☆ | 服务编排基本到位，但直接使用 ORM 对象绕过领域实体，事务控制隐含在 get_db 中 |
| 领域层 | ★★☆☆☆ | 形式上具备实体/接口/服务，但实体未被使用，仓储接口返回 ORM 类型违反 DIP |
| 基础设施层 | ★★★★☆ | FastCRUD 减少样板代码，多数据库支持良好；但缺少迁移工具，Redis 未集成 |

### 2.3 关键架构问题

#### 问题 1：领域实体形同虚设（严重）

**现状**：`domain/entities/` 定义了 `UserEntity`、`RoleEntity` 等 dataclass，ORM 模型也实现了 `to_domain()` / `from_domain()` 转换方法，但应用服务和仓储层全程直接操作 ORM 对象，领域实体从未被实际使用。

**影响**：
- 领域层的隔离价值丧失，业务逻辑可能泄漏到应用层或基础设施层
- 修改 ORM 模型会直接影响业务逻辑，失去 DDD 的防护作用
- 无法在纯内存中测试业务规则，必须依赖数据库

#### 问题 2：仓储接口违反依赖倒置原则（中等）

**现状**：`domain/repositories/` 的抽象接口返回类型为 ORM 模型（如 `User | None`），而非领域实体（如 `UserEntity | None`）。虽然通过 `TYPE_CHECKING` 避免了运行时导入，但概念上领域层依赖了基础设施层的类型。

**影响**：
- 接口的抽象性被破坏，切换 ORM 或存储方案时接口也要变
- 与 DDD 文档中声称的"领域层零外部依赖"原则矛盾

#### 问题 3：N+1 查询问题（中等）

**现状**：`require_permission(code)` 对用户的每个角色逐一查询菜单，5 个角色产生 6 次查询。

**影响**：权限检查是高频操作，每次请求都会执行，N+1 问题在大用户量下会显著影响性能。

#### 问题 4：缺少数据库迁移工具（严重）

**现状**：使用 `SQLModel.metadata.create_all()` 创建表，无 Alembic 集成。

**影响**：
- 无法做增量 schema 变更，生产环境升级必须手动写 SQL
- 无法回滚变更，数据丢失风险高
- 团队协作时 schema 变更无法版本化管理

#### 问题 5：Redis 已配置但未集成（中等）

**现状**：`RedisManager` 已实现，但无任何服务调用。Token 未做黑名单（logout 无效）、无缓存、无限流。

**影响**：
- Logout 后 token 仍然有效，存在安全风险
- 高频查询（如权限数据）无法缓存，数据库压力无法缓解
- 设置中的 `RATE_LIMIT_TIMES` / `RATE_LIMIT_SECONDS` 配置形同虚设

## 三、技术选型评估

### 3.1 核心框架评估

| 技术 | 优势 | 风险 | 大型项目适配度 |
|------|------|------|--------------|
| **FastAPI** | 异步高性能、自动文档、类型安全、生态活跃 | 无内建限流/Circuit Breaker、大型项目路由管理需额外设计 | ★★★★☆ |
| **SQLModel** | Pydantic+SQLAlchemy 融合、类型推导好 | 版本仍在 0.x、社区偏小、复杂查询受限 | ★★★☆☆ |
| **FastCRUD** | 减少样板代码、与 SQLModel 深度集成 | 灵活性受限、复杂场景需回退原生 SQLAlchemy | ★★★☆☆ |
| **classy-fastapi** | 类组织路由、结构清晰 | 社区极小、维护不确定、与 FastAPI 生态部分不兼容 | ★★☆☆☆ |
| **pydantic-settings** | 多环境配置、类型校验、.env 支持 | 功能相对简单、无配置中心集成 | ★★★★☆ |
| **python-jose** | JWT 标准实现 | 维护状态堪忧（长期未更新），建议迁移至 PyJWT | ★★☆☆☆ |

### 3.2 数据库选型评估

| 数据库 | 用途 | 评估 |
|--------|------|------|
| SQLite | 测试环境 | 适合单测，不适用生产 |
| MySQL | 开发环境 | 成熟稳定，适合中大型项目 |
| PostgreSQL | 生产环境 | 功能最丰富，适合大型项目复杂查询 |

**问题**：多数据库支持虽然灵活，但 SQLModel 的 `create_all()` 生成的 DDL 可能不兼容所有数据库方言，缺少迁移工具更放大了此风险。

### 3.3 异步架构评估

**优势**：
- 全链路异步（async def + AsyncSession + aiomysql/asyncpg/aiosqlite）
- 高并发场景下性能优势明显
- 与 FastAPI 原生异步模型一致

**风险**：
- 同步第三方库无法直接在异步上下文中使用，需 `run_in_executor` 桥接
- 调试和错误追踪比同步更复杂
- SQLAlchemy async session 的懒加载陷阱（已通过 `lazy="selectin"` 缓解）

## 七、改进建议

### 7.1 P0 — 必须修复（阻塞生产上线）

#### 1. 集成 Alembic 数据库迁移

```bash
cd service
alembic init migrations
# 配置 env.py 使用 SQLModel metadata
# 后续所有 schema 变更通过 alembic revision --autogenerate 管理
```

**收益**：生产环境安全升级、变更可回滚、团队协作版本化。

#### 2. 启用 Redis Token 黑名单

在 `AuthService.logout()` 中将 token 加入 Redis 黑名单，在 `get_current_user_id()` 中检查黑名单。

```python
# 伪代码
async def logout(self, token: str):
    payload = self.token_service.decode_token(token)
    expire = payload.get("exp", 0) - int(time.time())
    await redis.setex(f"token:blacklist:{token}", expire, "1")

async def get_current_user_id(self, token: str):
    if await redis.exists(f"token:blacklist:{token}"):
        raise UnauthorizedError("Token 已注销")
    # ... 正常验证
```

#### 3. 实现限流中间件

基于 Redis + 滑动窗口实现 API 限流，保护系统不被恶意请求打垮。

### 7.2 P1 — 应该修复（提升架构质量）

#### 4. 激活领域实体，实现真正的 DDD

- 仓储接口返回领域实体而非 ORM 对象
- 应用服务操作领域实体，通过 `from_domain()` 持久化
- 将业务规则（如密码策略、状态转换）封装到领域实体方法中

**改造步骤**：
1. 修改仓储接口签名，返回 `UserEntity` 而非 `User`
2. 仓储实现在 `get_by_id` 中调用 `model.to_domain()`
3. 仓储实现在 `create`/`update` 中调用 `Model.from_domain(entity)`
4. 应用服务使用领域实体，而非直接操作 ORM 对象

#### 5. 修复 N+1 权限查询

```python
# 优化前：N+1 查询
roles = await role_repo.get_user_roles(user_id)
for role in roles:
    menus = await menu_repo.get_role_menus(role.id)

# 优化后：单次 JOIN 查询
permissions = await menu_repo.get_user_permissions(user_id)
# SQL: SELECT DISTINCT m.name FROM sys_menus m
#      JOIN sys_userrole_menu rm ON m.id = rm.menu_id
#      JOIN sys_userinfo_roles ur ON rm.userrole_id = ur.userrole_id
#      WHERE ur.userinfo_id = :user_id AND m.menu_type = 2
```

#### 6. 启用 IP 过滤中间件

将 `IPFilterMiddleware` 注册到 `create_app()`，利用已有的 `sys_ip_rules` 表实现 IP 黑白名单。

#### 7. 优化批量删除

```python
# 优化前：逐条删除
for id in ids:
    await self.delete(id)

# 优化后：批量删除
await session.execute(sa_delete(User).where(User.id.in_(ids)))
```

### 7.3 P2 — 建议优化（长期演进）

#### 8. 替换 python-jose 为 PyJWT

`python-jose` 维护状态不佳，PyJWT 更活跃且更安全。

#### 9. 引入依赖注入容器

当模块超过 15 个时，手动 DI 工厂的维护成本会急剧上升。建议引入 `dependency-injector` 或 `lagom`。

#### 10. 引入事件驱动架构

- 领域事件解耦业务流程（如用户创建后发送通知）
- 为未来微服务拆分预留扩展点

#### 11. 添加 CI/CD 流水线

- GitHub Actions / GitLab CI 集成
- Ruff 检查 + MyPy 类型检查 + pytest 覆盖率门禁
- Docker 镜像自动构建与推送

#### 12. Docker Compose 补全

当前 compose 只包含 app 服务，应添加：
- PostgreSQL/MySQL 服务 + 数据持久化卷
- Redis 服务
- 健康检查依赖（app depends_on db/redis）

#### 13. 修复关联表命名

- `sys_userinfo_roles.userinfo_id` → `user_id`
- `sys_userrole_menu.userrole_id` → `role_id`
- 这需要一次数据库迁移来重命名列

#### 14. 统一审计日志事务

`RequestLoggingMiddleware` 应使用请求级别的 session 而非自建 session，确保审计日志与业务操作在同一事务中。

## 八、总结

### 整体评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ★★★☆☆ | DDD 分层结构在形式上到位，但领域层空转削弱了核心价值 |
| 技术选型 | ★★★☆☆ | FastAPI + SQLModel 适合快速开发，但部分依赖（classy-fastapi、python-jose）有风险 |
| 安全性 | ★★☆☆☆ | 认证基础在，但 Token 注销/限流/IP 过滤均未启用 |
| 可测试性 | ★★★☆☆ | 框架在，但仓储测试缺失、覆盖率不足 |
| 大型项目支撑 | ★★☆☆☆ | 缺少迁移工具、缓存、消息队列、CI/CD，距离生产级大型项目有差距 |

### 核心结论

当前架构是一个**结构良好的中等规模项目起点**，DDD 分层和模块化设计为演进提供了基础。但存在三个关键短板制约大型项目演进：

1. **领域层空转** — 最核心的架构问题，需要投入改造才能真正发挥 DDD 的防护价值
2. **运维能力缺失** — 无数据库迁移、无缓存集成、无 CI/CD，生产运维风险高
3. **安全机制不完整** — Token 注销、限流、IP 过滤均未启用

建议按 P0 → P1 → P2 优先级逐步改进，预计完成 P0 后可满足生产上线基本要求，完成 P1 后可支撑中型项目持续演进，完成 P2 后具备大型项目的架构基础。
