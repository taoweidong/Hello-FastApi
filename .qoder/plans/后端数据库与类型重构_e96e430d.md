# 后端数据库命名与类型一致性重构

## 背景

调研发现 service 后端存在以下问题：
- **12 张数据库表**均无 `sys_` 前缀
- **dept_id** 在 User 模型和相关 DTO 中定义为 `int`，但 Department 主键为 UUID `str`
- **日志 DTO**（LoginLog/OperationLog/SystemLog）的 `id` 定义为 `int`，但模型层为 UUID `str`
- **菜单 DTO** 的 `parentId` 存在 `str | int` 混合类型
- 部分代码可精简（如 menu_routes 手动创建 Repository 而非依赖注入）

## Task 1：数据库表名添加 sys_ 前缀

**目标**：将所有 12 张表的 `__tablename__` 添加 `sys_` 前缀

**涉及文件**：`service/src/infrastructure/database/models.py`

**变更清单**：

| 当前表名 | 新表名 |
|---------|--------|
| `users` | `sys_users` |
| `roles` | `sys_roles` |
| `permissions` | `sys_permissions` |
| `user_roles` | `sys_user_roles` |
| `role_permissions` | `sys_role_permissions` |
| `menus` | `sys_menus` |
| `departments` | `sys_departments` |
| `login_logs` | `sys_login_logs` |
| `operation_logs` | `sys_operation_logs` |
| `system_logs` | `sys_system_logs` |
| `ip_rules` | `sys_ip_rules` |
| `role_menus` | `sys_role_menus` |

**联动修改**：所有外键引用（如 `foreign_key="users.id"`）需同步更新为 `foreign_key="sys_users.id"`

## Task 2：修复 ID 类型不一致问题

**目标**：确保所有 ID 字段在模型层、DTO 层、服务层、API 层保持 `str`（36位UUID）类型统一

### 2a. 修复 User.dept_id 类型（int → str）

**涉及文件**：
- `service/src/infrastructure/database/models.py` — User 模型 `dept_id: int | None` → `str | None`
- `service/src/application/dto/user_dto.py` — UserCreateDTO/UserUpdateDTO/UserListQueryDTO 中 `dept_id` 和 `deptId` 从 `int` 改为 `str`
- `service/src/application/services/user_service.py` — 检查 dept_id 处理逻辑
- `service/src/api/v1/user_routes.py` — 检查 `dept` 字段转换逻辑（`dept_id or 0` 改为 `dept_id or ""`）

### 2b. 修复日志 DTO ID 类型（int → str）

**涉及文件**：`service/src/application/dto/log_dto.py`

**变更**：
- `LoginLogResponseDTO.id: int` → `str`
- `OperationLogResponseDTO.id: int` → `str`
- `SystemLogResponseDTO.id: int` → `str`
- `SystemLogDetailDTO.id: int` → `str`

### 2c. 统一菜单 parentId 类型

**涉及文件**：`service/src/application/dto/menu_dto.py`

**变更**：
- `MenuResponseDTO.parentId: str | int` → `str | None`
- 保留 MenuCreateDTO/MenuUpdateDTO 的 validator（前端可能传 int 0 表示顶级）

## Task 3：清理冗余代码与精简项目

**目标**：删除废弃代码，统一代码风格，精简项目结构

- 排查所有文件中未使用的 import 语句
- 检查并清理重复的验证器逻辑
- 清理 `.ruff_cache`、`logs/` 等无关文件（如有必要）
- 确保所有代码注释为中文

## Task 4：删除旧数据库并验证功能

**目标**：删除旧的 SQLite 数据库文件，重新启动服务验证所有功能

**操作**：
- 删除 `service/sql/dev.db`（旧表结构不兼容）
- 启动服务，验证数据库表自动创建
- 运行现有测试用例
- 运行 ruff/mypy 检查代码质量
