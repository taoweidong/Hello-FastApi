# 前后端服务联调修复与验证计划

## 问题总结

调研发现前后端存在以下关键不兼容问题：

| 问题 | 后端当前实现 | 前端期望 | 严重度 |
|------|------------|---------|--------|
| 响应成功码 | `code: 200` | `code: 0` | 高 - 导致所有请求被判定为失败 |
| Token 刷新路径 | `POST /refresh` | `POST /refresh-token` | 高 - Token 无法自动刷新 |
| 个人信息路径 | `GET /user/info` | `GET /mine` | 高 - 无法获取用户信息 |
| 动态路由接口 | 不存在 | `GET /get-async-routes` | 高 - 菜单无法动态加载 |
| 前端无 baseURL | - | 需设置 `/api/system` | 高 - 请求不经过代理 |
| Mock 拦截 | - | Mock 可能拦截真实请求 | 中 |
| 用户列表路径 | `POST /user/list` | `POST /user` | 中 |
| 角色列表方法 | `GET /role/list` | `POST /role` | 中 |

---

## Task 1: 修复后端 API 响应码格式

**目标**：将成功响应的 `code` 从 `200` 改为 `0`，对齐 pure-admin-backend 和 Pure Admin 前端的约定。

**修改文件**：
- `service/src/api/common.py` — 修改 `SuccessResponse` 等响应模型的默认 code 值为 `0`

---

## Task 2: 添加/对齐后端 API 路由

**目标**：添加前端期望但后端缺失的 API 端点，使后端路由与 pure-admin-backend 保持一致。

**修改文件与内容**：

1. **`service/src/api/v1/auth_routes.py`**：
   - 添加 `POST /refresh-token` 端点（复用 `/refresh` 的逻辑）
   - 或将 `/refresh` 重命名为 `/refresh-token`

2. **`service/src/api/v1/user_routes.py`**：
   - 添加 `GET /mine` 端点（在 auth_routes 或独立挂载，返回当前用户信息）
   - 添加 `GET /mine-logs` 端点（返回空列表或 stub 数据）
   - 检查用户列表接口路径是否需要同时支持 `POST /user`（前端调用方式）

3. **新建或在现有路由中添加** `GET /get-async-routes` 端点：
   - 根据用户角色返回可访问的菜单路由
   - 返回格式需兼容 Pure Admin 的 `asyncRoutes` 结构

4. **对齐其他系统管理 API**：
   - 确认 `POST /role`、`POST /menu` 等前端调用的路径在后端有对应实现
   - 检查 `/list-all-role`、`/role-menu` 等接口

---

## Task 3: 配置前端 API 基础路径与禁用 Mock

**目标**：配置前端 HTTP 客户端的 `baseURL`，使请求经过 Vite 代理到后端；确保 Mock 不拦截真实 API 请求。

**修改文件**：

1. **`web/src/utils/http/index.ts`**：
   - 在 axios 默认配置中添加 `baseURL: "/api/system"`
   - 这样前端 `/login` → 实际请求 `/api/system/login` → 代理到 `http://localhost:8000/api/system/login`

2. **`web/build/plugins.ts`** 或 **`web/.env.development`**：
   - 检查 Mock 插件配置，确保连接真实后端时 Mock 不会拦截 API 请求
   - 可通过环境变量控制 Mock 开关（如 `VITE_MOCK=false`）

3. **`web/vite.config.ts`**：
   - 确认 proxy 配置正确（已在上一轮修复）

## Task 4: 启动后端服务

**步骤**：
```bash

.venv\Scripts\activate
```
